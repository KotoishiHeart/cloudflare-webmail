# ADR 0007: Portable mailbox search

## Status

Accepted

## Context

The archived application used an FTS5 virtual table for subject, participant,
body excerpt, and attachment-name search. D1 supports FTS5, but D1 export does
not support databases containing virtual tables. The rebuilt system already
uses full portable D1 exports for backup, restore, and cutover verification.
The current production dataset is partitioned by mailbox in every user query,
so a mailbox-bounded scan is materially smaller than a global message scan.

## Decision

Use a normal strict table named `message_search_documents`, not a virtual FTS
table. One derived row contains up to 32 KiB of lower-cased subject,
participants (including outbound BCC), the bounded body preview, and attachment
filenames. D1 triggers create and maintain the row for normal writes, archived
bulk imports, and future repair operations.

The authorized message-list query always constrains `mailbox_id` and folder
before applying up to eight escaped search tokens. It also supports From,
To/Cc/Bcc, participant domain, JST date range, attachment/read/star tri-state,
size range, and the archived quick filters. Cursor pagination remains ordered
by `received_at, id`, so filter changes restart at the first page.

## Consequences

- Search keeps the same bounded-body behavior as the archived implementation;
  raw MIME and full R2 bodies are not copied into D1.
- Search documents are included in ordinary D1 backups and are regenerated by
  triggers during legacy bulk insertion.
- A very large single mailbox uses a bounded table scan instead of an FTS
  lookup. Query metrics must be reviewed after real production cutover; a
  separate regenerable search D1 can be introduced later without changing the
  primary backup format.
- Query text is always bound as data and escapes SQL `LIKE` wildcard syntax.

Cloudflare documents both [D1 FTS5 support](https://developers.cloudflare.com/d1/sql-api/sql-statements/)
and the [virtual-table export limitation](https://developers.cloudflare.com/d1/best-practices/import-export-data/).
